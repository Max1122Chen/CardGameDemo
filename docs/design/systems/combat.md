# 战斗系统

## 战斗循环

战斗以卡牌对战的形式，以“回合”作为时间单位。

玩家的战斗循环如此：

1. 开始时玩家从卡组中抽取若干张初始手牌；
2. 玩家每回合开始总是会获得一定的”费用“，用于支付打出手牌的消耗；
3. 玩家的回合开始时会从抽牌堆抽取若干张牌加入手牌；
4. 在玩家的回合中，玩家可以消耗费用以打出手牌或者使用背包中的道具，来对敌人造成伤害、进行防御、施展特殊效果；
5. 敌人在其回合会尝试攻击玩家或做出特殊行动；
6. 玩家的回合结束时，将手牌中的所有牌放入弃牌堆；
7. 当玩家尝试抽牌时，如果玩家的抽牌堆为空，则触发将弃牌堆洗牌后放到抽牌堆；
8. 当玩家或全部敌人一方死亡或达到特殊条件，或发生特殊事件，一局战斗结束；



进入战斗时，会根据玩家和敌人的速度或战斗前的关系决出先手权。



#### NPC

游戏中的敌人、中立、友好单位均统称为”NPC“，这是广义的NPC的含义，狭义上来讲NPC仅指那些能够与玩家”平和“交互的单位。

NPC和玩家一样都有属性系统中的各项属性。

##### AI

本质上讲，其他单位和玩家在战斗中可以做出的行动集合是相同的，它们背后也拥有一套卡组，并且在战斗时也遵守着抽牌、花费行动力行动的逻辑。

单位的AI可以分成几种：

- 固定型，单位总是会采取设计好的固定策略的行动而不会有“智能”，这种设计一般运用在设定上智力低下的单位，例如史莱姆，并且它们的卡组往往非常小且固定，使得每回合
- 智能型，单位会真正结合此时战斗状况和手牌情况来做出决策，一般运用在设定上具有智能的单位，例如人类，并且它们的卡组可能会有一定规模





## 战斗中的逻辑

战斗的各个环节用简单的语言就能描述并令人容易想象到情形，但实际上内在的逻辑要复杂得多，本章会讨论这些逻辑以及给出实现方案建议。



### 出牌、预计算与结算

在卡牌游戏中，计算卡牌的效用似乎不算复杂，只需要出牌时减去费用，并造成效果即可，事实也确实如此，但一个为了玩家体验的便利性措施为这个过程增加了复杂度。作为回合制、带有策略要素且数值计算相对复杂的卡牌游戏，我希望减轻玩家在脑海中苦苦计算一张牌的效用的困难，因此我希望玩家能够在选中将要出的牌和选中目标时，牌面上的伤害或者格挡的数值能够直接反映出牌后的结果。因此着就需要在玩家选中牌和敌人时就进行一次预计算，并在玩家真正决定出手时应用牌的效用。

但要如何实现这样一套“选目标预计算，出手后再结算”的机制？以下是我的答案：

拿最常见的伤害计算和伤害类型卡牌（攻击牌）举例，在属性系统的文档中，我们提出把“伤害”和“将要吸收的伤害”都作为一种属性，目的就在于这样能够在AttributeSet中“暂存”“伤害”这个概念，而不是立即对生命值进行修改。这种“临时存储等待结算”的属性在概念上被一些同行称为“元属性”。它的好处不仅仅是像在gameplay-framework中讨论的那样让描述对于一个属性的修改的复杂过程变得更好处理（伤害和将要吸收的伤害合在一起之于生命值就是对生命值的一种修改），而且分开了”计算“和”结算“，把它们分成了两个动作。

此外把“伤害”这个概念分离成攻击方的“伤害”受击方的“将要吸收的伤害”也是一个非常有意义的举措，实现了非常美妙的解耦。攻击方只需要计算出自己该造成多少伤害，受击方只需要拿到一个攻击方计算好的它声明自己会造成的伤害，然后计算自己应该真正吸收多少伤害即可，这把复杂的增伤减伤的逻辑和计算职责合理分离，良好地刻画了职责和权限边界。

在gameplay-framework中，我们提出用GFC这套系统塑造游戏逻辑，那么在这套系统的语境下，如何表达“出牌”的逻辑？

首先，“出牌”可以分成多个动作：

- 首先选择想出的牌
- 然后选择目标（有些牌有可能没有目标）
- 考虑是不是要出，这时数值部分的预计算应该已经完成了，玩家或者AI应该能够知道这张牌的效用是什么，这分为几个子步骤（假设是一张攻击牌）：
  - 出牌方计算自己将造成的伤害，修改Damage属性，这个动作完成之后触发目标的DamageToTake的**计算**
  - 目标的DamageToTake计算被触发，计算出自己的DamageToTake
- 根据出牌方的选择产生分支：
  - 选择出牌，那么因为预计算过程其实已经把应该造成的伤害算好了，直接触发目标的TakeDamage的动作，完成DamageToTake的**结算**之后出牌方和目标各自reset Damage和DamageToTake。
  - 选择不出牌（更换选中的牌，不选中牌，或是直接结束回合），类似地，reset Damage和DamageToTake即可。



用GameplayAbility的想法来看，把一张牌视作一个主动性的GA，把抽到一张牌看作当前这个GA处于可激活的状态，是自然的。但这里要立刻澄清一个容易混淆的点：出牌并不是在一个GA里从头到尾做完的。牌本身的GA主要表述的是“我有什么Effect”、选中之后如何完成预计算，以及之后如何等待外界的出手意图；真正把牌打出去这个动作，并不由这张牌的GA自己决定。

一个可行的两段式安排是这样的：选中牌（并且目标已经就绪时）激活牌的GA，先完成预计算（第一段），然后GA并不结束，而是进入一种监听状态。它监听是否出现TryPlayCard一类的事件，并且检查payload里的CardTag是否等于这张牌自己的Tag；若匹配，才真正尝试消耗行动力并走结算（此时才进入“出手”语义）。若始终等不到匹配的TryPlayCard，而是等到CancelPlayCard（换牌、取消选中、结束回合等），则把自己End掉，并在结束时重置刚才预计算挂上的临时数值。换句话说，Commit出牌是编排层通过事件发出的意图，牌GA只是响应；Cancel同理。

这样一来，“有这张牌的效果”和“打出这张牌”就拆开了：前者属于牌GA与GE；后者属于战斗会话（或输入层）对玩家/AI意图的表达。事件用在意图层（commit / cancel），而不是用来“触发一次数值计算”——数值计算仍然靠改属性和聚合器完成。

在上面我们说出牌方计算完自己的Damage（或称DamageToDeal）之后“触发”目标的DamageToTake的计算，这似乎有一种Trigger的意味，但实际上由于modifier聚合器的帮助，出牌方只需要给目标应用一个target.DamageToTake.BaseValue := src.Damage.CurrentValue（Override）的GE即可利用聚合器自动聚合出CurrentValue的效果让目标直接算出CurrentValue，而不需要真的通过传播事件来“Trigger”DamageToTake的计算。

类似地，我们称激活GA触发自己的计算其实也不算准确，真正的逻辑应该是在Damage的BaseValue初值为0的情况下，把卡面伤害数值Instantly加或者Override给Damage，这样聚合管线也能自动计算出CurrentValue，并不需要一个显式的动作“触发”这个过程。

Character的“计算将吸收的伤害”和“吸收伤害”这两个动作实际上可以各自统一抽象成一个GE和一个GA。

“计算将吸收的伤害”只需要给Character挂上一个Infinite的GE，把伤害吸收倍率，伤害吸收修正，伤害吸收偏移等聚合即可。这一步得到的是DamageToTake——它表达的是“将要吸收多少伤害”，而不是“将要掉多少生命值”。格挡并不参与这一层。

“吸收伤害”才涉及格挡结算，由TakeDamage一类GA完成：GA内采样格挡值和DamageToTake的当前值，算出两者差值，假设顺序是格挡先抵消将要吸收的伤害。

如果结果是正数，说明格挡大于将要吸收的伤害，那么只需要Apply一个Block -= DamageToTake的GE即可；

如果结果是负数，说明将要吸收的伤害大于格挡，那么要Apply一个HP -= (DamageToTake - Block)的GE和把Block清零的GE。

确认出手、完成结算之后，出牌方和目标各自reset Damage与DamageToTake；取消预览时同样只做reset，不改动已有生命与格挡存量。
