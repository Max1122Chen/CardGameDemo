## TL;DR

地牢里可点的东西（设施、NPC、机关脚本等）统一看成 **Interactable**。  
真正跨类型可复用的，主要是 **Dialogue + Options** 这一层会话壳，以及一套统一的交互接口。  
具体规则（泉水怎么回血、祭坛怎么献祭、乞丐怎么讨赏）由各自实现维护，不塞进一个万能的 Interaction 大类。

# 交互（Interaction）

本文讨论玩家在地牢探索中与“世界上的东西”发生的那一类交互：读一段说明、看到若干选项、做出选择、产生结果。

这里的 Interaction **不是** [gameplay-framework.md](./gameplay-framework.md) 里的 GameplayEvent。  
GameplayEvent 是实现层用来广播状态变化的机制；本文说的是玩法上“跟某个东西打交道”的体验设计。

此前在 [game-events.md](./game-events.md) 里用“游戏事件”举例（陷阱、机关等），强调的本质其实是：**给予玩家选项交互**。  
把乞丐、生命之泉、鲜血祭坛、遗弃熔炉等放在一起看会发现：它们不一定都适合叫“事件”，但玩家侧的外壳很像。更合适的顶层抽象是 **Interaction / Interactable**，而不是强行把所有东西都叫做游戏事件。

---

## 哪些东西算 Interactable

凡是玩家在探索中可以“对它做点什么”、并且通常伴随说明文字与选项的，都可以视为 Interactable，例如：

- **设施**：生命之泉、鲜血祭坛、遗弃的熔炉、休息用的篝火、上锁的箱子（若走选项而非单纯捡取）等等
- **NPC**：乞丐、商人（对话入口）、中立冒险者搭话等——强调“有人”，但交互外壳仍可与设施共用
- **脚本式遭遇 / 机关**：一次性陷阱提示、发现机关后的处理方式等——可以没有“人”，甚至没有长期存在的物件感，但仍走同一套交互入口

房间里还可能同时存在敌人、地上战利品等（见 [dungeon.md](./dungeon.md)）。  
**战斗确认、移动过门、结束探索轮、下楼撤离、单纯捡起地上物** 等，一般不先塞进 Interaction；它们已有或应有自己的流程。Interaction 管的是“跟某个 Interactable 开一次会话”。

---

## 不追求万能 Interaction 大类

一个容易走偏的做法是：设计一个巨大的 Interaction 类型，把泉水次数、祭坛献祭物、NPC 态度、陷阱 DC 等所有可能字段都堆进去，再用 `kind` 分支解释。

这样并不好：

- 大半字段对大半内容毫无意义
- 业务逻辑仍然不得不写在“解释器”的巨大分支里
- 名义上数据驱动，实际上既难读又难改

更符合直觉的做法是：

**独特的 Interactable 自己维护自己的状态与规则；统一的是接口与会话壳，不是字段大锅饭。**

Interaction 在程序上未必有很强的“一套配置打天下”的泛用性。很多逻辑依赖具体实现——泉水怎么算回复、祭坛吃什么、乞丐记不记得你施舍过——这些本来就该各自维护。

比较通用、值得单独抽出来的内核，主要是：

1. **Dialogue + Options**：当前要给玩家看的描述，以及可选的选项列表  
2. **统一的交互接口（API）**：主机（探索会话 / UI）只通过同一套入口开始交互、提交选项、结束交互，而不去理解熔炉内部有哪些字段  

需要共用的“小能力”（例如 [random.md](./random.md) 的骰子判定、扣血、发物品）可以做成工具被各实现调用，而不是继承成一个上帝基类。

## 概念关系（玩法视角）

可以粗略看成两层：

- **Interactable**：世界里“可交互的那一个东西”——挂在房间里的泉、站在房间里的乞丐、触发一次的机关脚本等。它负责自己的业务与状态（还能不能用、用过几次、态度如何）。
- **交互会话壳（Dialogue + Options）**：一次交互进行中，玩家正在看的文案与选项。Interactable 在需要说话或让玩家选择时，产出或推进这一层；玩家选定选项后，再交回该 Interactable 自己处理。

NPC 与“无人物件 / 脚本”的差别，主要在于 **身份与长期关系**（有没有人、会不会再遇见、要不要记态度），不在于是否使用 Dialogue + Options。

所谓“游戏事件”，在这个框架下更适合理解为：**某类 Interactable（或某次会话）的内容形态**，尤其是偏脚本、偏一次性、不一定强调“有人”的那一类；而不是与 Interaction 并列的另一套顶层系统。

---

## 和随机、属性、物品的关系

- 交互过程中常常需要判定（躲陷阱、劝说、赌运气）——使用 [random.md](./random.md) 描述的骰子规则，以及 [attributes.md](./attributes.md) 中的属性补正；判定是 **工具**，不是另一种 Interactable。
- 选项结果经常改生命、给物品、扣物品——走属性与 [items.md](./items.md) 已有语义，Interaction 不重新发明一套数值系统。
- 若将来被动装备要“在与泉交互时……”之类，可以通过 GameplayEvent 广播“开始/结束某次交互”一类信号；那是实现层挂钩，不改变本文对玩法交互的划分。

---

## 触发与生命周期（倾向，待实现时收紧）

设计上允许不同 Interactable 用不同触发方式，例如：

- 玩家在房间内主动选择“交互”
- 进入房间时自动弹出（更适合某些陷阱或强制遭遇）
- 休息等时机附带触发（见地牢中休息风险的叙述）

生命周期也可以各不相同：一次性消费、可重复使用、有使用次数、本层有效、跨层仍记得某些 flag 等。  
这些应由具体 Interactable 定义，而不是强行统一成同一种“事件消费”规则。

商店若以完整买卖界面为主，可以是独立子系统；也可以由某个 NPC Interactable 的选项“打开商店”。不必在第一版就把商店做成 Interaction 配置的一种字段。

## 举例（用来对齐直觉）

- **生命之泉**：设施；交互说明后选项可能是“饮用 / 离开”；饮用回血并可能耗尽或降效——状态在泉自己身上。
- **鲜血祭坛**：设施；选项可能涉及献祭物品或生命换取某种收益；规则与代价由祭坛自己维护。
- **遗弃熔炉**：设施；可能与装备或材料相关的特殊操作；同样是自己的业务。
- **乞丐**：NPC；对话与施舍选项；更强调“有人”，但呈现方式仍是 Dialogue + Options；是否再给报酬、是否记得你，由该 NPC 状态维护。
- **陷阱提示 / 机关**：可以没有长期“物件人设”，但仍通过同一套交互入口给出选项（小心通过 / 尝试拆除 / 强行踩过去等），内部用骰子判定。

这些例子说明的是 **外壳相似、内核（业务）各异**，而不是它们应共享同一张巨型属性表。

## 开放问题

1. 探索 UI 上，Interactable 是进房自动列出、还是需要玩家显式选中后再交互？  
2. 地上可直接捡的战利品，是否永远不走 Interaction，还是“打开箱子”类走 Interaction？  
3. 商人：对话树到什么程度结束、从哪一步交给专门的商店 UI？  
4. 简单对话树是否允许部分用数据（文案与选项）描述、复杂设施仍以具体实现为主？（倾向：允许混合，但不要求一切皆数据图。）
